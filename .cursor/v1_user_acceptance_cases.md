# V1 User Acceptance Cases

Главный критерий готовности `v1` для пользователя: не набор зелёных автотестов, а прохождение **10 живых сценариев** end-to-end без падений, с правильным поведением, понятным routing и корректным auto-resume/install flow там, где он нужен.

Автотесты и deterministic gates остаются обязательной страховкой, но **не заменяют** этот ручной прогон.

## Правило зачёта

`v1 ready` можно заявлять только если:

- все **10 из 10** сценариев ниже пройдены на живом runtime;
- gateway/UI/Telegram не падают во время прогона;
- бот не зависает в состоянии “надо что-то сделать вручную”, если сценарий ожидает auto-install, auto-create или auto-resume;
- routing выглядит разумным: простое идёт в дешёвый/локальный путь, сложное — в более сильный;
- после каждого сценария можно проверить, что именно произошло: ответ, артефакт, routing, install, resume, usage, inspector.

## Сценарии 1-8: Stage 86 foundation

Источник истины: [.cursor/stage86_test_cases.md](stage86_test_cases.md)

1. Простая реплика идёт в local route и отвечает быстро.
2. Сложный аналитический запрос идёт в stronger remote route и отвечает структурированно.
3. При отказе primary-модели fallback действительно продолжает выполнение без ручного спасения.
4. Запрос на отсутствующую capability вызывает install flow и после approval автоматически продолжается до результата.
5. Prompt optimization реально срабатывает и оставляет живой наблюдаемый след.
6. Runtime/session inspector показывает route/bootstrap/lifecycle truth.
7. Usage и cost видны в UI после живой сессии.
8. Hydra/SD1 direct route работает без auth/runtime сбоев.

## Сценарии 9-10: Investor-facing workflows

### 9. Сравнение двух прайсов

**Запрос пользователя:**

```text
Сравни эти два прайса и скажи, у кого лучше покупать.
```

**Что должен сделать бот:**

- принять два CSV/Excel файла без хрупкого ручного обхода;
- разобрать и нормализовать строки;
- сопоставить сравнимые позиции;
- выдать ranked summary table;
- дать внятную рекомендацию “покупать у X, потому что ...”.

**Что проверяем как человек:**

- [ ] файлы реально приняты и обработаны;
- [ ] бот не теряет контекст между файлами;
- [ ] ответ не распадается в сырой dump, а выглядит как осмысленная сводка;
- [ ] рекомендация опирается на данные, а не на пустую болтовню;
- [ ] если для вывода нужен более сильный маршрут, он реально используется.

### 10. Расчёт и отчёт по вентиляции

**Запрос пользователя:**

```text
Посчитай вентиляцию по этим размерам и сделай сводку. Если нужно, создай отчет.
```

**Что должен сделать бот:**

- принять размеры и ограничения;
- выполнить расчёт с assumptions, units и steps;
- использовать builder/project-designer context как лёгкий доменный слой, а не как отдельную тяжёлую capability;
- выдать структурированную сводку;
- при запросе на артефакт — создать markdown/PDF report без ручного развала сценария.

**Что проверяем как человек:**

- [ ] расчёт выглядит осмысленно и пошагово;
- [ ] единицы измерения и assumptions явно названы;
- [ ] бот не путает report-generation с chat-ответом;
- [ ] report действительно создаётся, если он был запрошен;
- [ ] маршрут выглядит умным: простые вычисления не уезжают без причины в самый дорогой путь, сложная сводка при необходимости усиливается.

## Что писать в финальном handoff

При заявлении `v1 ready` нужно перечислить:

1. Какие 10 сценариев были реально пройдены.
2. Какие артефакты/ответы были получены.
3. Где подтверждался routing/install/resume.
4. Какие ограничения остались, если они не ломают сценарии.

Если хотя бы один сценарий не пройден, `v1 ready` не заявляется.
